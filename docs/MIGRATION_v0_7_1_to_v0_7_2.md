# RiskRadar v0.7.1 → v0.7.2 마이그레이션

## 성격

v0.7.2는 **기존 신용 엔진 기록을 읽는 UI·표현 계층 업데이트**입니다.

다음은 바꾸지 않습니다.

- HY·BBB·A·CP 후보·확정 임계값
- `CreditEpisodeCfg`
- `credit_episode.py`의 상태 판정 규칙
- `decision_snapshot` schema
- aux schema
- FRED 수집 목록
- GitHub Actions workflow
- Telegram 전송 구조

## 새 화면

`기업 신용` 탭의 범위 지도 아래에 다음을 추가합니다.

1. **최근 90일 기업 신용 변화**
2. **지난 기업 신용 변화 기록 보기** 아코디언

타임라인은 `credit_episode_nodes.parquet`를, 과거 목록은 `credit_episodes.parquet`를 읽습니다. 새 데이터 아티팩트는 만들지 않습니다.

## 90일 정의

- 90개 관측치가 아니라 달력 90일
- 최신 신용 관측일을 종료일로 사용
- 구간 첫날 상태 전환 판정을 위해 노드별 직전 관측치 한 줄만 비교 문맥으로 사용
- 구간 밖 직전 관측치는 사건 목록에 표시하지 않음

## 후보 신호 표시 원칙

`estimated_onset`은 사용자에게 독립 사건으로 노출하지 않습니다.

확정되지 않고 사라진 `early_change`도 많을 수 있으므로, 확정된 에피소드의 후보 시작만 독립 성공 사례처럼 나열하지 않습니다. 상승 변화가 실제로 확인된 사건 아래에만 다음처럼 종속 표시합니다.

> 확인되기 전부터 2026-05-03에 초기 변화가 보이기 시작했습니다.

관측치 수는 설정 변경 시 달라질 수 있으므로 `2관측치 전`, `3거래일 전`처럼 고정 숫자를 쓰지 않습니다.

## 지난 기업 신용 변화 기록

지난 기업 신용 변화 기록은 append-only 영구 기록이 아닙니다. 최신 화면은 현재 이용 가능한 공식 원자료 범위에서 엔진을 다시 실행한 결과를 읽습니다.

따라서 사용자 문구는 다음으로 고정합니다.

> 현재 제공되는 공식 자료로 다시 계산한 기록입니다. 원자료 개정이나 이용 가능한 자료 범위 변화에 따라 과거 날짜와 변화가 기록된 시장이 조정될 수 있습니다.

`누적`이라는 표현은 사용하지 않습니다.

휴면 에피소드의 표시 종료일과 기간 계산은 같은 기준을 사용합니다.

1. `ended_at`
2. `dormant_at`
3. `last_meaningful_activity_at`

## 표현 정리

v0.7.0 표현 정리 패치에 남아 있던 기계 치환 흔적을 함께 수정합니다.

- `평소 변화가 나타난 곳` → `평소 범위`
- `자료 변화가 나타난 곳` → `자료 범위`
- `현재 변화가 나타난 곳` → `현재 변화 범위`
- `변화가 나타난 곳 엔진` → `기업 신용 범위 엔진`
- `신용 변화 흐름가`, `현재 변화가 나타난 곳와`, `데이터 데이터` 등 조사·중복 오류 제거

기존 기업 신용 상세 화면에서 `estimated_onset`을 `변화 시작`으로 직접 표시하던 부분도 제거하고 실제 `confirmed_at`을 `부담 상승 확인` 날짜로 표시합니다.

## 배포

전체 코드를 v0.7.2로 교체하면 됩니다.

- 새 secret 없음
- workflow 변경 없음
- 새 cache artifact 없음
- v0.7.1 데이터와 호환
- v0.7.0 데이터에서도 신용 기록이 있으면 타임라인 표시 가능

새 refresh는 필수는 아니지만, 배포 후 정상 refresh 1회로 코드 버전이 v0.7.2로 기록되는지 확인하는 것을 권장합니다.
